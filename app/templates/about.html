📘 About Denodo Docker Manager

```

Welcome to Denodo Docker Manager, a lightweight, powerful web application designed specifically for Denodo engineers, solution architects, and support teams. This platform simplifies and accelerates the way you spin up, manage, and experiment with Denodo Platform and database containers in a Docker-powered environment.

🔍 What Is This App?

  • Launch any Denodo version (7.0, 8.0, 9.0) with pre-configured roles (VDP, Scheduler, SM, etc.)
  • Auto-start popular databases like PostgreSQL, MySQL, and SQL Server
  • Manage all containers—running or stopped—in a unified UI
  • Simulate customer environments for bug reproduction or compatibility testing
  • Eliminate manual docker run / docker-compose complexity

🚀 Who Is It For?

  • Support Engineers: Reproduce bugs across multiple versions.
  • QA & Automation Engineers: Use consistent, reproducible setups.
  • Solution Architects: Demonstrate upgrade scenarios or mixed environments.
  • Denodo Developers: Prototype modules or external database integrations.

🧠 Why Not Just Use Docker Desktop or Rancher?

Feature Denodo Docker Manager Docker Desktop / Rancher
Denodo-centric UI ✅ Yes ❌ No
Launch Denodo roles easily ✅ Dropdowns + tags ❌ Manual setup
Docker Compose simulation ✅ Built-in ❌ Not supported
Logs/terminal UI ✅ One-click ❌ CLI required

🧰 Features At a Glance

  • Database auto-start via databases.yaml
  • Custom environment variable support for Kerberos, SSL, Auth
  • Import/export container configs
  • Real-time logs + terminal
  • Persistent volume paths for logs, drivers, config

🏁 Getting Started

Option A: Use startup.bat

```

@echo off
python -m venv venv
call venv\Scripts\activate
pip install -r requirements.txt
call fetch\_images.bat
uvicorn app.main\:app --reload --port 5665

Option B: Manual Run

python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main\:app --reload --port 5665
```

💡 Tips & Best Practices

  • Use autostart: true in databases.yaml to restart containers on app boot
  • Use terminal and log views to debug container boot issues
  • Mount JDBC drivers through volumes for custom support

🧪 Performance & Overhead

  • FastAPI async backend (~30MB memory)
  • Tailwind-based static frontend (no JS framework)
  • Startup under 1s (excluding image pulls)
```
